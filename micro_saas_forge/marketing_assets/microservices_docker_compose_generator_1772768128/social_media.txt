=== REDDIT POST ===
**I was so frustrated with manually wiring up multi-service Docker Compose files for local dev and staging environments, so I made a visual microservices Docker Compose generator.**

Every time I’d add a new service—say a Redis cache, a Postgres DB, and a couple of Node.js APIs—I’d waste an hour tweaking the docker-compose.yml: setting up networks, volumes, environment variables, health checks, and making sure the service discovery actually worked. Then I’d have to duplicate most of that logic into Kubernetes manifests later. It felt like busywork, and it was way too easy to introduce subtle config mismatches between environments.

So I built a tool that lets you define your services and dependencies in a simple declarative config (or drag them together in a visual workflow), and it spits out a production-ready docker-compose.yml along with Kubernetes manifests. It handles:

- Network topology and internal DNS
- Persistent volumes with sensible defaults
- Service discovery configs (env vars, etc.)
- Integrated local testing with simulated latency and failure injection (useful for testing circuit breakers)

The generator itself is built with TypeScript, Node.js, and React for the UI. The backend uses a custom YAML/JSON parser that maps to Docker Compose and K8s schemas, and the test environment leverages tc (traffic control) and some fault injection proxies we wrote in Go.

It’s something I built to scratch my own itch, but I’ve found it useful enough that I put it online for others: https://shipmicro.com/tools

Would love brutal feedback—especially if you’ve struggled with similar Docker/K8s wiring headaches, or if you think the approach is overkill. What’s your go-to method for managing multi-service setups locally?
